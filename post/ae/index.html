为什么要 Artifact Evaluation

省流：直接看这个 slide。

笔者最近经历了一次 Artifact Evaluation 的过程，心得这里记录一下。

对于理论性学科，比如数学，有了惊天的结论，当然是可以只通过论文来达到作者的效果。但是对于一个高度 empirical 和 experimental 的领域里，evidence 是起码。没有 evidence，根本谈不上什么贡献不贡献。对于计算机领域，evidence 就是代码。Talk is cheap, show me the code.

没有开源代码或被人实现的论文，默认就是怀疑它有问题。给部分代码的论文，默认它部分实验有 trick，敢直接给出完整代码的，那才是真正的 solid。所以从侧面来说，不给代码确实给你省时间了，因为你可以考虑不看。你的数据或者代码除非是商业机密（其实不建议你来发论文），效果是显著的话，作者是非常期待向全世界展示他的结果的，而不是遮遮掩掩。

近期，对于被接收的论文出了一项政策 Artifact Evaluation。在 ACM、IEEE、USENIX 等组织的会议上，论文作者在提交论文的同时，还能够提交 AE（Artifact Evaluation），由独立的第三方对论文代码或系统的功能完备性、结果的可复现性等进行评估，以检验论文的结论有数据和实验的支持。一旦作者完成 AE，则可获得对应的徽章并在论文首页展示。

如今，越来越多的作者选择参加 AE 并在首页贴上徽章。以 SOSP 2021 为例，有 40 篇论文（占总数 74%）提交了 Artifact ，包括软件实现、硬件实现、测试脚本和数据集等用于研究或实验的材料。其中，39 篇论文的系统是 “可获得的”，38 篇论文的系统通过“功能完备” 的评审，27 篇论文的结果通过了 “可重现” 的评审。除 SOSP 外，OSDI、PPoPP、ASPLOS 等计算机系统相关的会议都陆续增加了 AE 评审。

Collberg 等研究者统计了 2011 年到 2013 年期间 8 个计算机学术会议及 5 个计算机期刊的论文代码可获取与可复用的情况，在排除需要特殊硬件（HW）、论文结果不需要代码支撑（NC）、论文作者存在冲突（EX）等论文之外，有近一半的论文无法提供源码（176/402，EM）。在能够提供源代码的工作中，只有 60% 的工作在论文发表后提供了公开可获取的源码（Article+Web），而剩下的 40% 是通过发送邮件向作者请求源码后获取。而在最终获取源码的这些工作中，有近 40% 的工作要么需要超过 30 分钟构建（OK>30），有些作者声称在有限的修改后可以构建（OK Auth），甚至还有 9 份代码连原作者也承认构建中存在问题。这还仅仅只是在构建代码层面，而代码对应的论文中数据与结论是否能够由第三方在另外的系统中进一步复现就更不得而知了。在此背景下，独立的、用于评估论文实现以及结果可复现的 AE 流程应运而生。

然而，引入 AE 的过程并非一帆风顺，很多人对于该流程存在不少顾虑。首先，部分研究者认为如果强制进行 AE，会对于论文作者不公平，毕竟作者可能耗费很长时间与人力资源才完成项目的工程构建。其次，有些研究者认为构建方便的可复用代码可能会耗费大量时间，而这样做能否带来足够的收益也不是很明确。此外，还有的项目可能是与工业界合作，项目代码涉及保密及所有权的问题。最后，如果某个会议强制要求 AE，可能会导致论文的投稿数量下降。

考虑到上述顾虑，当前的 AE 流程普遍采用了以下原则。首先，大部分的 AE 与论文是否收录无关；即无论 AE 通过与否，都不会影响论文的接收。因此，即使 AE 评委认为提交的 Artifact 无法复用，或无法在 AE 评委的环境中复现论文的结果，都不会导致论文被拒收。其次，大部分会议并不强制进行 AE 。对于工作涉及保密的工作，作者可以选择不进行 AE，或者进行 AE 但是选择 Artifact 不公开（即放弃 Available 认证）。在不选择 Available 认证的前提下，作者仍可以请求 AE 的评委对于论文的结果进行复现，提升数据与结果的可信度，而不必担心保密的代码被公开。

通过这一系列措施，目前的 AE 流程对于不同类型的工作都十分友好，在考虑到不同工作的特殊性的前提下，尽可能地鼓励各位作者通过 AE 为社区快速发展作出贡献。

AE 的结果一般由徽章体现，不同的徽章表明提交的 AE 材料通过相应的审核。常见的徽章包括：可获得、功能完备、结果可复现，等等。

AE 使用的一些术语：

tip

  • Repeatability：可重复性 （相同的团队，相同的实验设置） 同一团队使用相同的测量程序、相同的测量系统，在相同的操作条件、相同的位置进行多次试验，可以获得规定精度的测量结果。对于计算实验，这意味着研究人员可以可靠地重复他自己的计算。
  • Reproducibility：可复现性 （不同的团队，相同的实验设置） 可以由不同团队使用相同的测量程序、相同的测量系统，在相同的操作条件、相同或不同的位置进行多次试验，可以获得规定精度的测量结果。对于计算实验，这意味着一个独立的小组可以使用作者提供的材料获得相同的结果。
  • Relicability：可复制性 （不同的团队，不同的实验设置） 可以通过不同的团队、不同的测量系统，在多次试验的不同位置，获得规定精度的测量结果。对于计算实验，这意味着一个独立的小组可以使用他们完全独立开发的系统获得相同的结果。

AE 的徽章：

tip

  • Artifact Available：代码或系统可获得（绿色） 需保证相关的代码或系统可被公开获取。比如保存在 Zenode, FigShare, Github, Gitlab 等平台，注意不能使用个人网页。
  • Functional：功能完备（浅红色） 需保证代码或系统在功能、可用性和相关性方面是否与论文一致，即这些代码或系统是否有效，能否支撑论文中的结果。具体会考虑以下三个方面： 文档：备注是否足够详细，使读者能够正常使用？ 完整性：提交的代码或系统是否包括论文中描述的所有关键组件？ 可执行性：提交的代码或系统是否包含运行论文中描述的实验所需的脚本和数据，软件能否成功执行？
  • Reusable：可重用（深红色） 可重用是功能完备的进阶版。功能完备中仅包括了能复现功能所需最少的文档等材料。在可重用中，需要包含非常完善的文档、使用手册、结构化的代码，能有效促进其他开发者重用该代码或系统。实际 AE 过程通常不包含这一点。
  • Results Replicated：结果可复制（浅蓝色） 第三方的团队需要通过作者提供的材料，独立获得论文中提出的主要结果。在实际 AE 评审过程中，往往使用结果可复现来代替此徽章。
  • Results Reproduced：结果可复现（深蓝色） 提交的代码或系统必须能够获得论文中提出的主要结果。简而言之，第三方团队是否有可能独立重复实验并获得支持论文的结论。值得注意的是，该徽章的目标不是准确再现结果，而是在允许的容差范围内独立生成结果，以便验证论文的结论。

观察与经验：

观察 - 1：大部分 AE 评委还是会选择在提供的机器上进行快速实验 / 复现。本文作者进行 AE 的两个工作的 6 个 AE 评委，均选择连接到提供的测试机器进行测试。当然，这一方面与论文工作都与具体类型的硬件相关，但经过与其他完成 AE 的同学交流，这具有一定普适性。因此，在提供机器上能否无障碍地快速实验就非常关键。

经验 - 1：在提交 AE 文档之间，最好提前登录提供的 AE 账号，按照给出的手册重新完整运行一次进行检查，从而避免例如权限、文件绝对路径等问题。

观察 - 2：根据 AE 期间的反馈，AE 的评委还是非常欢迎这种最直接也是最便捷的方式进行复现。

经验 - 2：对所有的实验，提供 click-to-run 脚本，其包含编译、测试、收集数据、画图等功能。同时，这些脚本的内在逻辑需要非常清晰，AE 评委能够快速根据脚本理解各个部分的用途，以及找到对应源码的位置进行深入阅读。

经验 - 3：可以通过样例输出对结果的清晰说明、解释。说明文档可以包含样例输出，以及输出对应到论文具体的表格与图，以及能够支撑具体第几页第几章第几行的结论，方便 AE 评委能够快速理解实验的数据，判断是否一致。

经验 - 4：对所有的测试缩小测试规模，并写明如何在更大规模 / 重复测试、以及测试所需要的时间。AE 的评委并非需要一个绝对稳定的数据（往往通过多次重复实验取平均实现），因此我们在提交的脚本中，最好默认采用一个较小的测试规模（即次数），从而尽可能缩小评估所要时间。但同时要说明如何进行更大规模（测试次数）测试，比如：“修改脚本中第几行的 TIMES 变量值来修改每个测试执行数”。此外，在说明文档里面包含每项测试或每个单元测试需要的时间，可以进一步减少 AE 评委在测试时的负担。

artifact evaluation 的目的不是确认论文或者代码是 “对” 的，也不是要做 code review，而是为了确认论文的 claim 和代码实现是 consistent 的。如果论文 claim 了错的东西，但是代码和它是 consistent 的，从 artifact evaluation 的角度看这个 artifact 是没问题的。目前 artifact evaluation 的实践确实就是要把代码跑一遍或多遍。对于有误差的部分，论文的作者有义务给出复现时可接受的误差范围。至于随机化参数的问题已经有一些解决方法了，比如 deterministic container 的技术，会将随机参数的种子确定下来，以便复现，虽然这么做也只是增加了一定程度的 reproducibility，但总是比没有要好的。

有时候我觉得 AI/ML 这个社区真的是很奇怪。一方面搞出来 http://OpenReview.net 和 JMLR 这样的 publication mode，在我看来对于 open science 是非常进步的事情，但另一方面面对 reproducibility 的危机，对于在 PL/SE/System 领域已经广泛采用的 artifact evaluation 模式似乎毫无兴趣。比如按照 ACM SIGPLAN 使用的模式，论文被接受后会被邀请提交 artifact（通常以配置好的虚拟机 / Docker image 的形式提交），这个 artifact 会被一个独立的 evaluation committee 来验证其是否可以支撑论文中的 claim。代码能正常工作的话授予一个 artifact functional/reproducible 的章，代码完全开源并且有 DOI 的话再打一个 artifact available 的章，代码写得好有文档有注释再打一个 artifact reusable 的章。当然 artifact 不是评价文章的唯一标准，但有这些章无疑会增加文章的说服力。

不是所有文章的模型都是 google 级别的，对于非常大的模型，也可以退而求其次，要求作者公开存档 artifact，以后有人有资源和机会再来验证。至于所谓 “极端情况”，打死也复现不出来这个模型效果，那么这种文章很可能连起码的 scientifically sound 的要求都没有达到，为什么要发表呢？没有统一的运行环境、各种随机参数和初始值、超参数、巨大的模型，这些正是产生 reproducibility 危机的原因。AI / ML 社区需要做的正是解决这些问题。

思考

  • 精力守恒：假如一个研究者，连续几年有明显超过同机构 PhD 几倍的一作者论文量，那么其工作的复现性，一般较低。这是因为有深度的和扎实的工作往往需要非常多的时间来想出方案和反复论证。
  • 价值传承：假如一个学生师成于作风踏实的导师，那么他传承的 “科研价值观” 一般都会比较踏实。我自己发现的一个有趣的现象是：假如作者或者其导师在欧洲受过学术训练，那么其工作都是相对好复现的。
  • 环境所迫：我一般对于某些地区的论文会特别谨慎。这些地区的共同特点是：科研评估指标非常强调论文数量（这些地区应该不难猜到，我就不点破啦）。在 Tenure Track 和 Peer Pressure 的双重作用下，研究人员不得不批量制造论文，不少论文只有故事，缺乏反复论证，可复现度相当低。
